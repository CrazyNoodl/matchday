## What the app is

1-vs-1 football/FIFA tournament tracker for small groups. Players compete in round-robin rounds, stats are recorded per match, standings are calculated with H2H tiebreaker. Cloud sync via Supabase.

Platforms: iOS, Android, Web. Expo SDK 56, React Native 0.85.3, React 19.2.3.

## State model

```
matches          â€” current open round only (cleared on finishRound)
archivedRounds   â€” past rounds of the current active tournament
closedTournaments â€” fully finished tournaments (hasTournament = false after closeTournament())
```

- Once `closeTournament()` fires, `hasTournament` â†’ false, matches move into `closedTournaments`, all edit UI disappears.
- Stats screen (`app/stats.tsx`) aggregates ALL three layers: `closedTournaments` + `archivedRounds` + `matches`. The merge/union/H2H-pair logic behind this (`collectAllMatches`, `collectPlayerIds`, `buildH2HPairs`, `sumGoals`, `countMatchDaysPlayed`) is extracted into `src/utils/statsAggregation.ts` â€” pure functions, unit-tested in `src/utils/__tests__/statsAggregation.test.ts` â€” rather than left as inline `useMemo` blocks in the screen, specifically so the 3-layer merge and the H2H perspective-flip logic have real regression coverage. `app/season-stats.tsx` (single-`ClosedTournament` scope, no cross-layer merge) has its own `filterRoundsByRanked`/`countChampDaysWon` extracted the same way into `src/utils/seasonStatsAggregation.ts`, tested in `src/utils/__tests__/seasonStatsAggregation.test.ts`. **Known quirk, not fixed, flagged for a future call**: `countChampDaysWon` counts any round (including a friendly one) whose `winner` matches `champId`, but the champion itself is decided from ranked rounds only (`closeTournament()` in `tournamentSlice.ts`) â€” so with the season-stats INCLUDE filter set to "Friendly" or "Both", a champion's friendly-round wins can inflate "match days won" beyond what crowned them.
- Modal system is a discriminated union in `src/store/types.ts`; all modals rendered inline in their screen, driven by `store.setModal('name')`.
- `deletePlayer` (`src/store/slices/playersSlice.ts`) also prunes the deleted id from `tournamentPlayers`/`roundPlayers` â€” otherwise a deleted player who'd already been removed from their match leaves a ghost zero-stat row in `app/tournament.tsx`'s standings (which trusts `tournamentPlayers` as source of truth, no filtering against `players`).
- `deleteTeam` (`src/store/slices/teamsSlice.ts`) also no-ops when any `players[].teamCode` still references the team, not just when the team appears in matches (#82) â€” mirrors the match/archive/closed-tournament guards `deletePlayer` already had. Without this, deleting a team that was only ever a player's *default* team (never actually played a match) left that player with a dangling `teamCode`, breaking `TeamBadge`/`Avatar` color lookups wherever the player is shown.
- Round ordinals are **not** read from a stored field at display time. `startRound()` still writes `ArchivedRound.n`, but `src/utils/roundOrdinals.ts` (`getRankedRoundOrdinals`) recomputes the ranked-only ordinal live from `archivedRounds.filter(r => r.ranked)` every render â€” friendly rounds consume no slot and show `â€“` instead of a number. This makes historical data self-correct without a migration if the numbering rule ever changes again.
- `StatKey = KnownStatKey | (string & {})` (`src/store/types.ts`) is deliberately **not** used as a `Record`'s key type (only at single-value-field positions) â€” TS makes every literal member of a union-with-generic-branch a _required_ property when it's a `Record` key, which breaks sparse maps like `Match.statsOverride` where only some of the 23 stats may be set.
- Two intentionally different tie-handling behaviors coexist in `src/utils/standings.ts`, don't conflate them: `calculateStandings`' sort order (via `applyH2HSort`) is a **fully deterministic total order** â€” pts â†’ GD â†’ GF â†’ H2H (pts/GD/GF) â†’ total wins â†’ stable `playerId` hash â€” so `standings[0]` is always a single, reproducible winner; this is what `closeTournament()` (`tournamentSlice.ts`) uses to crown the tournament champion, by product decision a tournament must always have exactly one. `isTopTied()` is a separate, narrower check (pts/GD/GF/H2H equal only â€” no wins/hash fallback) used solely to detect a genuine draw for a single **round's** `winner` field (`buildArchivedRound` in `tournamentSlice.ts`, `app/round.tsx`'s finish-round flow) â€” a round is allowed to end in a recorded draw (`winner: ''`, no banner shown) even though the tournament as a whole never is.

## Storage folder structure

Supabase Storage (`match-media` bucket) layout for new uploads:

```
{userId}/{tournamentId}/matchday-{date}_{HHmm}/match_{aScore}-{bScore}_{date}_{HHmm}/{timestamp}-{randomId}.{ext}
```

- **Round folder**: generated once by `startRound()` (`buildRoundFolder()`), lives on `TournamentState.roundFolder` while open, copied to `ArchivedRound.folder` by `finishRound()`.
- **Match folder**: generated once at match creation (`buildMatchFolder`), stored on `Match.mediaFolder`. **Fixed at creation, never renamed** â€” editing the score or re-scanning stats later does not rename the folder.
- `matchMediaFolder(roundFolder, match)` (`src/store/sliceHelpers.ts`) is the single shared helper every upload/delete path uses to resolve a match's folder â€” no `match.mediaFolder` â†’ falls back to `match.id` (pre-folder-structure legacy match); folder but no `roundFolder` â†’ bare match folder; both present â†’ `${roundFolder}/${match.mediaFolder}`. Both the Add Match flow and the match-detail "+ Add" flow must go through this helper â€” they diverged once before (see below) and produced inconsistent paths for the same match.
- Team logos upload to a flat `{userId}/team-logos/...` path â€” unaffected by the round/match hierarchy above.
- **Cleanup is prefix-based**: `deleteMatch`/`deleteRound`/`deleteArchivedRound`/`deleteClosedTournament` each delete their whole folder in one `deleteStorageFolder()` sweep (recursive `list()`â†’`remove()` walk â€” Supabase Storage has no native recursive delete) rather than per-item. Legacy matches with no `mediaFolder` are cleaned up via an individual fallback delete since they don't live under a round-folder prefix.

## Gotchas & non-obvious behavior

### Sync & auth

- **Persistent auth session**: `src/supabase/client.ts` builds an MMKV-backed `auth.storage` adapter for native (a separate MMKV instance, `supabase-auth`, from the main store's `matchday-store`) â€” Supabase's native default is `@react-native-async-storage/async-storage`, which isn't installed in this project, so without this the session only ever lived in memory and every cold restart dropped back to login. Web keeps Supabase's default `localStorage` adapter.
- **Sign-out clears local state**: `confirmSignOut` (`src/screens/settings/useSettings.ts`) calls `store.resetStore()` **before** `signOut()` (order flipped 2026-07-15, fixing #80) â€” otherwise a second account signing in on the same device would see the previous account's cached `players`/`teams`, and the sync bootstrap-push could push that stale data into the new account. Wiping local state first means a crash between the two steps just leaves the user still signed in as the old account with local data intact (safe to retry) instead of a cleared session paired with stale data that the next account could inherit. `resetStore()`'s cloud-media deletion (`deleteMediaItem()` sweep over every match photo/player photo/team logo URI) is an explicit opt-in, `resetStore({ deleteCloudMedia: true })` â€” only "Reset All Data" passes it. Sign-out never did, but before 2026-07-09 it relied on an accident of call order (by the time `resetStore()` ran, `signOut()` had already cleared the session, so `deleteMediaItem()`'s own `getCurrentUserId()` returned `null` and it silently no-op'd) rather than an explicit guarantee â€” see the [Demo Mode incident](#demo-mode-leaked-into-supabase-2026-07-09). See [Account-switch data leakage](#account-switch-data-leakage--fixed-2026-07-15) for the full fix, including local backup scoping and account-scoped display prefs.
- `public.players`' primary key is a composite `(id, user_id)` (migration `supabase/migrations/003_players_composite_key.sql`, matching `teams`' `(code, user_id)`) â€” **must be applied manually** against the live Supabase project, it does not run automatically.
- **Persisted pending-sync + reconnect retry** (`src/supabase/useSyncManager.ts`, `src/store/index.ts`): dirty-table tracking is mirrored from the in-memory `dirtyRef` into a persisted store field, `pendingSyncTables`, via `persistDirty()` â€” otherwise a crash/force-quit before the debounced push fires would silently lose the edit (nothing would re-mark it dirty on next launch). On launch and on reconnect, a push always runs before any pull, since `applyCloudState()` is a blind overwrite that would wipe unsynced local edits with a stale cloud snapshot.
- **`useIsOnline()` must be called once at the app root and threaded down as a value, not re-invoked per feature.** Each call site mounts its own independent NetInfo/window-event listener and its own `pingSupabase()` health-check cycle with its own state. `useMediaRetryManager` originally called `useIsOnline()` itself instead of taking `isOnline` as a param â€” two concurrent health-check fetches to the same URL (one from the root, one from this hook) made the *root's own* `isOnline` reading get permanently stuck reporting "unreachable" in the e2e offline test (`e2e/09.offline.spec.ts`, which routes-and-aborts all Supabase network calls): the offline banner stayed visible for the whole test timeout instead of clearing once verified reachable. Passing the root's already-computed `isOnline` down (same pattern `OfflineBanner` already used) fixed it. Exact mechanism not fully root-caused (suspected browser-level interaction between two simultaneous aborted fetches to the identical URL), but empirically confirmed: removing the duplicate hook instance alone made the test pass reliably.
- **Supabase reachability health-check** (`src/supabase/health.ts`): NetInfo/browser online-events only report "is there a network route," not "can we reach our backend" â€” a phone with data cut off for non-payment, or a captive-portal wifi, still reports "connected." `pingSupabase()` hits GoTrue's `/auth/v1/health` and `useIsOnline()` layers this on top of the raw signal (checked periodically, and on `AppState` â†’ `'active'`). Final `isOnline = rawOnline && hasVerifiedOnce && !verifiedUnreachable` â€” `hasVerifiedOnce` withholds "online" until the first ping for the current `rawOnline` signal has resolved, so a wifi-up-but-no-WAN connection doesn't briefly read as online for the ~1-2.5s the ping takes to time out. One shared hook, so every offline-gated button and the sync reconnect trigger benefit automatically.
- **Per-feature offline gating**: media upload/delete, OCR import, and team logo picker are all disabled while offline (`!useIsOnline()`) rather than letting the user hit a confusing failure later. Deliberately **not** gated: login/sign-out (blocking them would remove the only recovery path), and dev-only screens.
- Web's `useIsOnline` deliberately bypasses NetInfo and listens to `window`'s `online`/`offline` events directly â€” NetInfo prefers the Network Information API on web when available, and that event does not reliably fire on a real connectivity drop in Chromium.
- **E2E fixtures block real Supabase network calls** (`e2e/fixtures.ts`'s `blockSupabaseNetwork`): `authedPage`/`mockSharedRoundRpc` route-and-abort all requests to the project's `*.supabase.co` host as a hard guarantee, so no test can reach the shared dev/prod Supabase project. `playwright.config.ts`'s `webServer` used to pass empty `EXPO_PUBLIC_SUPABASE_*` env vars intending `supabaseConfigured=false` for the whole e2e run, but Expo/Metro inlines env vars from `.env` directly at bundle time and ignores that override â€” the real project URL/key were baked into the web bundle on any machine with a local `.env` regardless. This was silently inconsistent with CI, which has no `.env` at all: there, the empty override actually took effect, so `supabaseConfigured` was `false` in CI but `true` locally.   - **That guard didn't actually work, and this was the bigger find (2026-07-22).** The app registers a Service Worker (`public/sw.js`, `app/_layout.tsx`) for offline caching; once active, a page's `fetch()` is intercepted by the SW's own `fetch` event handler, which re-issues the request from the *SW's* execution context. `page.route()` only hooks the page/frame's network stack, not a Service Worker's â€” so `route.abort()` never actually reached the SW's inner fetch, and `blockSupabaseNetwork` silently blocked nothing whenever the SW was active. Every e2e test that ever ran with `supabaseConfigured=true` (i.e. every local run, since local always has a real `.env`) was making genuine live requests to the production Supabase project with `FAKE_SESSION`'s unsigned JWT (rejected 401/403 by the real server â€” confirmed via a real Playwright trace, not synthesized â€” not mutating data, but a real, unintended live dependency this fixture's own comment already worried about without actually preventing). Fixed with `use: { serviceWorkers: 'block' }` in `playwright.config.ts` â€” no e2e test needs the SW's caching behavior, and blocking registration outright removes the whole bug class.
  - Blocking the SW this way also fully engaged `blockSupabaseNetwork` for the first time â€” which broke `useIsOnline()`'s reachability ping (`pingSupabase`, `src/supabase/health.ts`): it hits the same `*.supabase.co` host with a harmless unauthenticated `/auth/v1/health` GET, previously always leaking through to a real 200 (silently keeping every test "online"). Fixed by adding a second, more specific route in `blockSupabaseNetwork` that fulfills `/auth/v1/health` with a fake `200` instead of aborting it (registered after the wholesale block â€” Playwright routes are matched most-recently-registered-first). Explicit offline tests (`page.context().setOffline(true)`) are unaffected: that flips the browser's own online/offline signal, which `useIsOnline()` trusts ahead of ever reaching the ping.
- Not-logged-in + offline â†’ full-screen `OfflineScreen` replaces `LoginScreen` (no local-first path without a session). Logged-in + offline â†’ a small non-blocking `OfflineBanner` (`pointerEvents="none"`, since a purely informational banner must never intercept taps underneath it) â€” the app stays fully usable with local data. Offline boot deliberately does **not** clear the store or force sign-out.
- Local JSON backup (`/settings/backup`) auto-pushes to Supabase immediately after the single "Replace all local data?" confirmation â€” there is no separate "Push to Cloud" button/second modal anymore; that confirmation's copy now explicitly covers the cloud overwrite too, so one explicit tap authorizes both the local and cloud replace (still consistent with the "destructive/irreversible action needs its own explicit function" rule â€” restoring a backup is that one function, `handleConfirmImport` in `backup.tsx`, not a generic dirty-diff push). A full-screen blocking overlay covers the local-apply + cloud-push window (relevant for large backups over a slow connection). ### Media & storage

- **5-item media cap** per match, enforced both in the picker's `selectionLimit` and by disabling the add button.
- **Optimistic upload**: media is saved to the store immediately as `{ uri: localUri, uploading: true }`; replaced with the remote URL on success or `{ pendingUpload: true }` (retry overlay) on failure. Both `uploading` and `pendingUpload` items are **stripped on store rehydration** (`onRehydrateStorage`) so a crash mid-upload doesn't leave stuck spinners on next launch.

It now marks the item `pendingUpload: true` (same shape match-detail already used), so the existing retry overlay on the match detail screen picks it up automatically once the match is saved and opened â€” no separate UI needed.
- **Video is temporarily disabled** ([#59](https://github.com/CrazyNoodl/matchday/issues/59)): both pickers request `mediaTypes: ['images']` only. Video items already attached to older matches are hidden from display (`useMatchDetail.ts`'s `visibleMedia` filters them out, preserving original array indices for delete/retry) but **not deleted** â€” they still count toward the 5-item cap. Root cause not yet investigated.
- Image resize presets (`src/utils/imageResize.ts`, via `expo-image-manipulator`): `MEDIA_MAX_DIMENSION` 2000px (regular media), `OCR_PAYLOAD_MAX_DIMENSION` 2000px (base64 sent to AI), `STAT_PHOTO_STORAGE_MAX_DIMENSION` 1200px (persisted stat screenshot), `TEAM_LOGO_MAX_DIMENSION` 600px. ### Stats / OCR

- **All 23 stats always render in a fixed canonical order** (`buildMergedStats()`, `src/utils/mergedStats.ts`), regardless of what OCR captured. A key missing from `statsOverride` renders as a muted "0" (`isNA: true`), not a distinct "N/A" string. Unknown OCR keys are appended after the 23, marked `isCanonical: false` â€” only those non-canonical rows can be deleted from the edit sheet (the 23 canonical ones are edit-only, never removable except via the sheet's global "Clear").
- `Match.statsOverride` entries carry optional `confidence?: StatConfidence` (shown as a small yellow dot when `'low'`/`'medium'`); confidence is dropped the moment the user manually edits that row, since it's then user-confirmed.
- `adjustStat` rounds to 1 decimal after every +/- (avoids float drift like `2.0999999...`) and clamps `[0, 100]` for percent stats. `touchedStats` tracks which params were edited in the current session â€” an untouched param stays excluded from the saved override even if its displayed value is 0.
- **Per-photo OCR validation gate**: a photo must recognize â‰¥8 of the 23 canonical stat keys (`MIN_CANONICAL_STATS`) or it's treated as the wrong screenshot â€” excluded from the merge entirely, never added to `match.media`, but still uploaded to Storage with a `rejected-` filename prefix for later cleanup (no cleanup job exists yet). Re-scanning with only an invalid photo leaves existing stats/media untouched.
- **`StatsRow`'s `aWins` is tri-state** (`boolean | null`, `null` = tie) ([#69](https://github.com/CrazyNoodl/matchday/issues/69)): both call sites (`app/match/[id].tsx`, `app/settings/(developer)/ocr-lab.tsx`) compute `stat.aVal === stat.bVal ? null : stat.aVal > stat.bVal` â€” a plain `boolean` with a `>=` comparison silently made every exact tie render as an "A wins" highlight. `null` renders both bars/values as neutral gray in `StatsRow.tsx`.
- `extractStatsFromPhoto` drops AI-returned keys that just duplicate the match score (`goals`, `score`, `finalScore`, `matchScore`, `result`) before they reach the merge step, and the OCR prompt explicitly tells the model not to return them.
- **Per-photo OCR tracking, not a flat merged blob** ([#71](https://github.com/CrazyNoodl/matchday/issues/71)): both the Add Match flow (`src/utils/addMatchState.ts`'s `ocrPhotos: OcrPhotoEntry[]`, replacing the old flat `ocrAssets`) and OCR Lab (`app/settings/(developer)/ocr-lab.tsx`'s `PhotoItem.stats`) track each photo's own scan result independently instead of only the final merged result. This fixed a real bug in the _production_ Add Match flow (not just the OCR Lab dev tool the issue was filed against): adding a new photo used to re-send every already-scanned photo to the AI provider again, and removing any photo after a scan wiped every other photo's already-good stats, forcing a full re-scan. Now adding only scans the new photo(s), and removing a photo drops just its own contribution â€” `pendingStats` (Add Match) / `stats` (OCR Lab) are derived by merging each photo's own result (`src/utils/ocrPhotoMerge.ts`'s `mergeStatArrays`/`toPendingStatsRecord`, shared by both screens). A photo that fails OCR no longer nukes a sibling photo's already-succeeded stats â€” `ocrStatus: 'error'` isolates to the specific failing photo, and removing that one specific photo self-heals the status back to `'done'`.

### UI components & theming

- **`MediaSlider` touch architecture â€” do not regress**: the outer `Pressable` (closes on tap outside) must wrap `FlatList` directly, never with another `Pressable` in between â€” a wrapping `Pressable` claims the touch responder before `FlatList`'s pan handler can negotiate, killing swipe. Each slide is its own `Pressable` inside the list (standard RN pattern); the pagination dots need `position: 'absolute'` or they get pushed below the full-height `FlatList`, off-screen.
- **Reordering a `Gesture.Pan()` row's array position while its gesture is still active silently cancels that gesture** (found building `DraggableMatchBlock`'s drag-to-reorder, see the "What is fully implemented" entry): calling the React state setter that reshuffles the `.map()`'d array â€” even just once, mid-drag â€” moves the actively-dragged row's DOM node to a new sibling position, which browsers/gesture-handler treat as the pointer being lost, silently ending the gesture (confirmed via full stack trace: `useState`'s reducer â†’ the store action â†’ `scheduleUpdateOnFiber`, not a guess). Symptom was oddly specific and easy to misdiagnose as something else: every continuous drag capped at exactly one swap, "working again" only after releasing and re-pressing (a fresh gesture on the now-repositioned node). ## Sync data-loss incident (2026-07-06)

**Production incident:** a user's real Supabase data (players, teams, active tournament, matches) was permanently deleted. No Supabase backup/PITR existed (Free plan) â€” unrecoverable. Two independent bugs in the sync layer were found and fixed:

1. **`resetStore()` cascaded into a real cloud delete.** `resetStore()` (called by both "Reset All Data" and sign-out) wiped local Zustand state; `useSyncManager`'s subscriber saw `players`/`teams`/tournament fields change, marked them dirty like a normal edit, and the debounced push fired with an emptied payload â€” `pushState()` deletes any cloud row missing from the payload, i.e. it deleted every row for that user. **Fix:** a `syncSuppressionRef` (`src/store/index.ts`) is set around `resetStore()`'s wipe; the sync subscriber bails out early while it's set.
2. **Realtime-triggered `pull()` race.** After any push, a realtime event schedules a `pull()` ~400ms later. It unconditionally applied whatever `pullState()` returned without re-checking dirty/pushing state after the network `await` â€” a team or player added during that window got silently overwritten by the stale pre-add snapshot. **Fix:** `pull()` now bails out if `pushingRef.current || dirtyRef.current.size > 0` at the moment it's about to apply.

**Design correction found mid-fix:** "Reset All Data" is _intentionally_ a full wipe including cloud data â€” making `resetStore()` unconditionally local-only broke that intent. The correct shape: destructive cloud deletion must only ever happen via one explicit, directly-called function, never as a side effect of generic dirty-diffing. `deleteAllCloudData()` (`src/supabase/sync.ts`) is that function; `handleReset` calls it explicitly before `store.resetStore()`. Sign-out still only does the local-only wipe.

**Structural risk still open:** this project uses a single Supabase project for both local development and real user data (every worktree gets the same `.env` copied in) â€” a dev-time mistake hits real data directly, as happened here. Worth a separate dev/staging Supabase project.

This incident is also why the [local JSON backup](#what-is-fully-implemented) feature exists â€” a second, independent safety net entirely separate from Supabase (`src/utils/backup.ts`, `/settings/backup`). Its `applyBackupLocally()` is bracketed with the same `syncSuppressionRef`. Backups deliberately exclude **all** media links (player photos, team logos, match photos/videos â€” `buildBackupPayload()`'s `stripPlayerMedia`/`stripTeamMedia`/`stripMatchMedia`/`stripRoundMedia`/`stripClosedTournamentMedia`): those are Supabase Storage URLs, a backup can be restored long after it was made, and a "Reset All Data" in between would leave any embedded URL pointing at nothing. Restoring a backup never reintroduces a broken media reference â€” the tradeoff is that photos/logos/match media are never restored either and must be re-added manually.

## Overlapping-push race deleting matches added while offline (2026-07-09)

**Reported symptom:** add matches to an open round while offline, force-quit the app, reconnect, relaunch â€” the offline-added matches were gone. Staying inside the app through the same offlineâ†’online transition never lost anything.

**Root cause:** `useSyncManager.ts`'s store-subscribe listener rescheduled a debounced `runPush()` on every qualifying edit without checking whether a push was already in flight (`pushingRef.current`). If a second edit's debounce settled while an earlier push was still mid-network-call (offline, slow connection, retry), both pushes ran concurrently, each with its own `buildSyncPayload()` snapshot taken at its own start time. `pushState()`'s `openMatches` section is upsert-current-list-then-delete-by-absence â€” if the _earlier_ push (stale, smaller matches array) finished its network round-trip _after_ the _later_ push (fresher, bigger array) had already landed, the earlier push's delete-by-absence removed the newer match from the cloud even though it had just synced successfully. Because both pushes reported success, `dirtyRef`/`pendingSyncTables` ended up empty â€” so on the next relaunch, `init()` skipped the "flush leftover dirty tables" step and went straight to `pull()`, which blind-overwrote local state with the now-incomplete cloud snapshot, deleting the match locally too. Staying in the app didn't hit this because the reconnect path's `pull()` is guarded by `pushingRef`/`dirtyRef` checks that a fresh relaunch doesn't retain.

**Fix:** added `pushQueuedRef` (`src/supabase/useSyncManager.ts`). A debounce that settles while `pushingRef.current` is true no longer starts a second overlapping push â€” it just sets `pushQueuedRef.current = true` and returns. `runPush()`'s `finally` checks that flag and, if set, immediately calls itself again with a fresh `dirtyRef`/state snapshot (which by then includes every edit made during the first push's flight). Regression test: `src/supabase/__tests__/useSyncManager.overlappingPush.test.ts` â€” uses a stateful fake Supabase client (real upsert/delete/select semantics, not scripted responses) with manually-held promises to force the exact interleaving. `src/supabase/__tests__/useSyncManager.offlineMatchLoss.test.ts` covers the adjacent (already-fixed) force-quit-with-a-single-edit case for comparison â€” both now pass.

## Demo Mode leaked into Supabase (2026-07-09)

**Reported symptom:** a demo tournament (`demo-tour-pl-s2`, "Premier League S2" â€” the hardcoded id in `src/demo/data.ts`'s `DEMO_STATE`) was found in the real Supabase `tournaments` table via the Table Editor, sitting next to the user's actual active tournament under the same `user_id`.

**Root cause:** `setDemoMode(true)`/`setDemoMode(false)` (`src/store/slices/settingsSlice.ts`) swap `players`/`teams`/`matches`/tournament fields wholesale between `DEMO_STATE` and `realDataBackup` in one `set()` call. `useSyncManager.ts`'s dirty-tracking subscriber compared references unconditionally and treated that swap as a real edit, persisting it into `pendingSyncTables` â€” even though entering demo mode itself correctly skipped scheduling a push. If the app was then force-quit while Demo Mode was still on, the next cold start's leftover-dirty flush (`init()`) called `runPush()` with **no `demoMode` check anywhere in that path** â€” pushing the live (still demo) state straight to Supabase under the real `user_id`. The same false-dirty flag also permanently broke the exit-demo mode "pull latest cloud state before overwriting it" safeguard: `pull()` bails out whenever `dirtyRef` is non-empty, and the restore itself always left it non-empty, so that pull always short-circuited to `'failed'` and the safeguard never actually ran.

Two adjacent, previously-unnoticed gaps were found investigating this:

1. **`resetStore()`'s cloud-media deletion was unconditional.** It swept every match photo/player photo/team logo URI (from both live state *and* `realDataBackup`) and called `deleteMediaItem()` on each â€” correct for "Reset All Data" (`deleteAllCloudData()` already wipes the DB rows, so wiping Storage too is intended), but sign-out called the same `resetStore()` with no way to opt out. In practice sign-out never actually deleted anything, because `confirmSignOut` calls `signOut()` first, which clears the session `deleteMediaItem()`'s own `getCurrentUserId()` call depends on â€” so it silently no-op'd. That protection was incidental (dependent on call order), not explicit.
2. **Photos/logos picked while Demo Mode was active uploaded to the user's real Supabase Storage.** Demo Mode only swaps local state â€” the Supabase session stays fully valid, and none of the five upload call sites (`useAddMatchFlow.ts`'s add-match flow, `useMatchDetail.ts`'s add-media/OCR-import/retry-upload, `teams.tsx`'s logo picker) checked `demoMode`. A photo added to a demo match uploaded for real, then got orphaned forever once Demo Mode exited and discarded the local reference via the `realDataBackup` restore.

**Fix (`fix/demo-sync-leak`):**

- `runPush()` (`src/supabase/useSyncManager.ts`) now refuses to push whenever `demoMode` is true, regardless of caller â€” a last-line-of-defense guard, not just a fix at each call site.
- The dirty-tracking subscriber no longer marks anything dirty on a demo-mode enter/exit transition (detected directly via `state.demoMode !== prevState.demoMode`, replacing a `prevDemoModeRef` that could itself go stale). This is also what un-breaks the exit-demo pull-before-push safeguard.
- `resetStore()` (`src/store/index.ts`) takes an optional `{ deleteCloudMedia?: boolean }` â€” only `handleReset` passes `true`. Sign-out's `resetStore()` call (`src/screens/settings/useSettings.ts`) no longer attempts cloud-media deletion at all, closing the latent risk regardless of call order.
- The five upload call sites now check `demoMode` and keep the picked photo/logo local-only (skip the Supabase upload entirely) while Demo Mode is active â€” the edit is discarded on exit anyway, so there was never anything meant to persist in the cloud.
- The Developer menu row (`app/settings/index.tsx`) is hidden while Demo Mode is on, and the Developer screen itself (`app/settings/(developer)/developer.tsx`) redirects back if `demoMode` becomes true while it's mounted â€” its tools (Import Round, OCR Lab, Resize Lab) write into whichever tournament is currently active, which would silently be the demo data.

Regression tests: `src/supabase/__tests__/useSyncManager.demoModeSyncLeak.test.ts` (leftover-dirty flush never pushes while `demoMode` is true; exiting demo mode actually applies a newer pull instead of always short-circuiting) and `src/store/__tests__/resetStoreCloudMedia.test.ts` (`deleteCloudMedia` opt-in, including the demo/`realDataBackup` double-jeopardy case). Both confirmed to fail against the pre-fix code before being kept.

**Cleanup still needed:** the orphaned `demo-tour-pl-s2` row (and any demo player/team rows it pulled in) must be deleted manually from Supabase via the Table Editor â€” this fix prevents recurrence, it does not retroactively clean up data already leaked.

## Account-switch data leakage â€” fixed (2026-07-15)

Investigated 2026-07-14, fixed 2026-07-15 in `fix/account-scoped-data-leak`. Filed as three separate GitHub issues after a codebase investigation prompted by a user-reported symptom (restore-backup menu showing the previous account's backup after switching accounts on the same device). All three shared one root cause pattern: state that should be scoped per-account was actually scoped per-device (or not scoped/cleared at all), because `resetStore()` (`src/store/index.ts`) â€” sign-out's only cleanup step â€” only cleared the Zustand/MMKV store, not everything else that lives on the device.

1. **[#79](https://github.com/CrazyNoodl/matchday/issues/79) â€” Local backups leak across accounts. Fixed.** `listBackups()`/`createBackup()` (`src/utils/backup.ts`) now tag backups with the signed-in user's id and filter the list to the current user; sign-out clears the previous account's backups instead of leaving them listed/restorable for the next account on the same device.
2. **[#80](https://github.com/CrazyNoodl/matchday/issues/80) â€” Non-atomic sign-out could push account A's data into account B's cloud rows. Fixed.** `confirmSignOut()` (`src/screens/settings/useSettings.ts`) previously awaited `signOut()` then separately awaited `store.resetStore()` â€” if the app was killed between the two, the next account to sign in could trigger `useSyncManager.init()`'s bootstrap-push path against account A's still-present local data (since `applyCloudState()` preserves local state on an empty cloud pull, treating stale leftover data as a legitimate first sync). Sign-out is now atomic from the store's point of view, so a mid-sequence crash can no longer leave stale account data behind with a cleared session.
3. **[#81](https://github.com/CrazyNoodl/matchday/issues/81) â€” Display prefs were device-scoped, not account-scoped. Fixed.** `showNick`, `showTeamLogo`, `groupByTours`, `showAvgGoals`, `standingsViewMode`, `language` (`colorScheme` deliberately excluded â€” kept device-scoped, since a user may want dark mode on one device and light on another) now sync through the same account-scoped settings layer as the rest of the app's data (`supabase/migrations/005_user_settings.sql`, `src/supabase/sync.ts`), instead of living only in the local store's `partialize`.

Grep-confirmed no other unscoped local storage exists outside these three â€” `backup.ts`, the main store, and `src/supabase/client.ts` (the latter is correctly session-scoped and cleared by `signOut()`).
